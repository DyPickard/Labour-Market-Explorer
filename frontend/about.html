About this project

A full-stack dashboard that visualises Statistics Canada Labour Force Survey data for Canada's 65 economic regions. A Python pipeline pulls live government data into a database, a Flask API serves it, and a component-based front end renders it as interactive, hand-drawn SVG charts. Every figure comes from real data — nothing is mocked.

Python pandas SQLAlchemy SQLite Flask requests JavaScript React SVG HTML / CSS StatCan WDS API
How it fits together
StatCan WDS API → pipeline.py (ETL) → SQLite → api_payload.py → Flask · /api/labour → Browser (SVG dashboard)
The data
  • Source: Statistics Canada's Labour Force Survey, pulled from the public Web Data Service (WDS) REST API — so the numbers are the same ones StatCan publishes.
  • Three tables: labour-force characteristics by economic region, monthly (14-10-0462); employment by industry, annual (14-10-0023); and employee wages by industry, annual (14-10-0064).
  • An honest caveat: labour-force figures are published per economic region, but industry and wage data are only published per province. Those panels therefore show the selected region's province, and are labelled that way rather than pretending to be regional.
The data pipeline

pipeline.py asks the WDS API for each table's current download link, streams the zipped CSV, and unzips it in memory — no files touch disk. pandas then filters the rows that matter (economic regions, the latest reference periods, the relevant characteristics), cleans and reshapes them, and SQLAlchemy writes the result into a local SQLite database.

Because it reads straight from the live API, the whole pipeline can be re-run whenever Statistics Canada releases new monthly data to refresh the dashboard.

The backend

api.py is a small Flask app. It serves the front-end files and one JSON endpoint, /api/labour, which returns every region's data in a single response.

api_payload.py builds that response by reading the SQLite tables and assembling, for each region, its monthly rate series, industry breakdowns, and labour-force composition. The payload is cached in memory so repeat requests are instant, with a refresh endpoint to rebuild it after the pipeline runs.

The front end

The interface began as a self-contained design component (an HTML template rendered by a lightweight React runtime) that shipped with placeholder data. The core of the work was replacing that mock data with the real thing: the component fetches /api/labour once on load and renders the entire dashboard client-side from the response.

Every chart is drawn by hand as SVG — the KPI sparklines, the dual-axis unemployment/participation trend line with hover tooltips, the horizontal industry and wage bars, and the donut charts. There is no charting library. The styling was pulled out into a single stylesheet of semantic classes, and the layout is a responsive CSS grid.

Details worth calling out
  • 65 regions, 15 years of history: every economic region has its own monthly series driving the KPIs, trend line, donuts, and ranking.
  • A layout that never jumps: the region picker is a fixed width and the header truncates gracefully, so choosing a region with a very long name doesn't reflow the page.
  • Plain-language tooltips: every tile has an info icon that explains what the statistic actually measures.
  • Charts that tell a story: the two donuts read as a funnel of the workforce — the labour force splitting into employed and unemployed, and the employed splitting into full-time and part-time.
  • Accuracy over polish: figures are labelled honestly wherever the data differs from the ideal — provincial industry data, three-month moving averages, and the latest available reference period.
